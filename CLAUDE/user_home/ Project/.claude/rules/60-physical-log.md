# 60-physical-log.md - 電池急減調査用物理ログ開発ルール

**最終更新**: 2026-04-16  
**バージョン**: 2.2

## 1. 最優先原則（必須）

- 既存の動作確認済みコードは原則変更しない。
- 本フェーズは観測目的。バグを見つけても修正せず記録する。
- 新規実装より既存コード活用を優先する。
- 診断処理が製品動作へ影響しないことを最優先する。

## 2. 実装制約

- 新規実装は `src/physical_log/` に配置する。
- 既存コードへの変更は最小フック追加のみ許可。
- 動的メモリ確保は禁止。静的確保のみ。
- RAM使用量は必要最小限（目安: 70KB以下）。
- サンプリング間隔は 30秒固定、処理時間は軽量化（目安: 10ms以下）。

## 3. 許可フック

- `drv_overcurrent_sens.c` -> `physical_log_motor_event()`
- `drv_gear_unlatch.c` -> `physical_log_gear_unlatch_event()`
- 初期化系 -> `physical_log_init()`

上記以外のフック追加は事前承認必須。

## 4. バグ発見時の対応

- 保存先: `docs/調査/`
- ファイル名: **JST `YYYYMMDD_hhmmss_` プレフィックス必須**
  - 例: `20260416_102530_バグ報告_power.md`
- 記載: 概要 / 発見場所 / 現象 / 推定原因 / 影響範囲 / 推奨修正案 / 備考
- 本ルール内でバグ修正PRやワークアラウンド実装は行わない。

## 5. 優先順位

1. 既存システム安定性の維持
2. 観測データの取得
3. 実装のシンプルさ
4. 効率化

## 6. 追加ルール（P-P）

- P-P1: `[TL]` ログ契約（キー/単位/順序）をFWとツールで同期する。
- P-P2: G1（モータ区間残り値）とG2（真のIDLE）を混同しない。
- P-P3: BAT24残留は電流センサのみで断定せず、GPIO直接観測を優先する。
- P-P4: D1/D2/D3 はワークキュー経由で実行される前提で解析する。
- P-P5: 切り分け順序（A1/A2 -> E/F時系列）を固定する。
- P-P6: 累積カウンタは絶対値でなく差分で分析する。

## 7. レビュー観点

- 既存変更が最小か（フック限定か）
- ログ追加が非破壊か（優先度/タイミングを壊していないか）
- メモリ・CPU負荷が許容範囲か
- バグ発見時に「修正ではなく記録」を徹底したか
