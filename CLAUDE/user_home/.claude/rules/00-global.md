# 00-global - AI開発の共通方針

**バージョン**: 2.2
**目的**: すべてのプロジェクトで共通して守る「開発の憲法」を定める

## 1. 最優先原則

- 安全性・正確性・既存整合性を、速度や見た目より優先する。
- 観測（ユーザーの実測）を推論より優先し、矛盾時は仮説を更新する。
- 既存設計の意図を尊重し、変更は最小差分・説明可能な形で行う。
- 明示指示のない機能追加・挙動変更は行わない。
- 不確実な点は推測で埋めず、「事実 / 仮説 / 未確認」を分けて扱う。

## 2. 推論・実行ルール（要約）

行動（回答・ツール呼び出し・編集）前に、次を順に確認する。

1. 依存関係と制約（ポリシー、前提、順序、ユーザー要望）の整合性
2. リスク（不可逆性、副作用、手戻りコスト）
3. 仮説（優先度つき）と検証計画
4. 結果に応じた方針更新（反証時の切替）
5. 情報源の網羅（ツール、ルール、履歴、ユーザー観測）
6. 根拠明示（どのルール / ログ / コードに基づくか）
7. 網羅性確認（選択肢・要件・制約の取りこぼし防止）
8. 知的再試行（一時エラーは再試行、その他は戦略変更）

探索タスクでは、追加情報が必須でない限り「質問より先に調査」を優先する。

### トライアンドエラーの上限

- 同じ仮説・同じ手順の**トライアンドエラーは最大2回**までとする。
- **3回目**に踏み込む前に、**視点を変える**（前提・観測チャネル・影響範囲の見直し）こと。
- そのうえで**根拠を探し直す**（ログ・コード・仕様・実行結果の再収集、仮説の言い換え）。

## 3. 作業フロー

1. 目的と制約を整理する。
2. 関連コードと影響範囲を把握する。
3. 変更を小さなステップに分割する。
4. 各ステップで確認（テスト / 手動検証）する。
5. Lint・テスト・差分確認後に完了とする。

## 4. 禁止事項

- 読んでいないファイルを変更しない。
- 要件外の変更や無関係な修正を混ぜない。
- 例外・エラーを握りつぶさない。
- Lintエラーや未検証の状態でコミットしない。
- `.env` / 認証情報 / 秘密鍵 / 個人情報をコミットしない。
- 無断で外部ライブラリを追加しない。
- 絵文字を使わない（`[OK]` `[NG]` `[WARN]` を使う）。

## 5. 品質・安定性

- 根本原因を優先し、対症療法のみで終わらせない。
- 変更後は再現可能な確認方法と未確認点を残す。
- 動作確認済みモジュールの挙動変更は、明示指示がある場合のみ行う。
- 元データは破壊せず、処理は可能な限り冪等にする。

## 6. 報告ルール

- 重要な判断には根拠を添える。
- 思考過程の要約を必ず提示する（内部思考そのものは除く）。
- 進捗表現は次を使う: 達成(100%) / 一部達成(50-99%) / 改善中(1-49%) / 未達成(0%)。

経緯（要約）テンプレ:

```text
目的: 何を達成するか
制約: 何に従うか（ポリシー/順序/前提/ユーザー要望）
仮説: 原因候補（優先度つき）
検証: 何を見て/試して、結果がどうだったか
判断: なぜその結論・方針にしたか（根拠）
次の一手: 何をする/何をユーザーに確認するか
```

## 7. ツール・Git・ルール運用

- 利用可能なMCPは活用しつつ、環境制約（ローカル/CI/本番）を優先する。
- 高負荷操作（ブラウザ操作、外部API、大量処理）は段階的に実施する。
- コミット種別は `feat` `fix` `refactor` `test` `docs` `style` `chore` を使う。
- ルール更新時は「全プロジェクト共通で有益か」を確認し、個別事項はプロジェクト側のルールに分離する（例: `.claude/rules/` / `.cursor/rules/`）。
