# Modularization Rules

## 基本方針

- テスト工数は運用コストの支配要因。境界は再テスト負荷の最小化を目的に引く。
- 抽象化の根拠は測定可能な運用利得のみ。「設計の美しさ」は根拠にしない。
- 分離コスト（ファイル +1、import +1、学習 +alpha）は有限。根拠なき分離は負債。

## 分離を正当化する3つの根拠

いずれか1つ以上を明示できるときに分離する。

### 根拠1: 再利用（2+ callers）

- 今日2箇所以上から使われていれば切り出す。
- 「そのうち使うかも」は不可（YAGNI）。

### 根拠2: 変更隔離（blast-radius containment）

- 大きな安定モジュール内に「将来独立に変わる見込みの1箇所」があれば、1 caller でも切り出す。
- 変更時に周囲の再テストを省略できるなら、分離コストは回収できる。
- 例: 外部 API のセレクタ/フィールド名、UA、エンドポイント定数、フィーチャーフラグ、モック差し替え境界。

### 根拠3: 純粋ロジックのテスタビリティ

- I/O を伴う orchestrator から純粋関数を抽出し、単体テスト可能にする。
- 2+ callers がなくても妥当。

## 正当化されない分離

- 「concerns が違う」だけを理由とする分離
- 起きていない仮説への先行対応（Protocol / registry / DI / version 切替）
- 物理的に割るだけで callers も isolation も変わらない分離
- clean architecture の形式論のみに基づく階層化

## 判断手順

1. 今日 2+ callers があるか
2. 将来変わる見込みの「1箇所」を周囲の再テストから守りたいか
3. 純粋ロジックを I/O から切り離してテスト可能にしたいか

1つでも Yes なら分離候補。全て No なら埋め込んだままにする。

## 提案時に明示する項目

- 分離対象（ファイル / 関数 / 定数）
- 該当根拠（1/2/3、複数可）とその具体
- 根拠1: callers の列挙
- 根拠2: 想定変更の種類・頻度・影響範囲
- 根拠3: テスト可能になる振る舞い
- 分離コスト（追加ファイル数、影響 import 本数）

## 既存モジュールの再評価

- callers が1に減り変更も止まったら統合検討
- 埋め込み部分の変更頻度が上がったら切り出し検討
- 変更のたびに「根拠2で切り出すべきだったか」を自問する
