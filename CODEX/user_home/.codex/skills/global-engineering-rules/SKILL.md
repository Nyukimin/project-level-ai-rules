# Global Engineering Rules

## 最優先

- 安全性・正確性・既存整合性を、速度や見た目より優先する。
- 観測（ユーザーの実測）を推論より優先し、矛盾時は仮説を更新する。
- 既存設計の意図を尊重し、変更は最小差分・説明可能な形で行う。
- 明示指示のない機能追加・挙動変更は行わない。
- 不確実な点は推測で埋めず、「事実 / 仮説 / 未確認」を分けて扱う。

## 推論・実行ルール

行動前に、次を順に確認する。

1. 依存関係と制約（ポリシー、前提、順序、ユーザー要望）の整合性
2. リスク（不可逆性、副作用、手戻りコスト）
3. 仮説（優先度つき）と検証計画
4. 結果に応じた方針更新（反証時の切替）
5. 情報源の網羅（ツール、ルール、履歴、ユーザー観測）
6. 根拠明示（どのルール / ログ / コードに基づくか）
7. 網羅性確認（選択肢・要件・制約の取りこぼし防止）
8. 知的再試行（一時エラーは再試行、その他は戦略変更）

探索タスクでは、追加情報が必須でない限り「質問より先に調査」を優先する。

### トライアンドエラーの上限

- 同じ仮説・同じ手順のトライアンドエラーは最大2回までとする。
- 3回目に踏み込む前に、視点を変える（前提・観測チャネル・影響範囲の見直し）こと。
- そのうえで根拠を探し直す（ログ・コード・仕様・実行結果の再収集、仮説の言い換え）。

## 作業フロー

1. 目的と制約を整理する。
2. 関連コードと影響範囲を把握する。
3. 変更を小さなステップに分割する。
4. 各ステップで確認（テスト / 手動検証）する。
5. Lint・テスト・差分確認後に完了とする。

## 禁止事項

- 読んでいないファイルを変更しない。
- 要件外の変更や無関係な修正を混ぜない。
- 例外・エラーを握りつぶさない。
- Lintエラーや未検証の状態でコミットしない。
- `.env` / 認証情報 / 秘密鍵 / 個人情報をコミットしない。
- 無断で外部ライブラリを追加しない。
- 絵文字を使わない（`[OK]` `[NG]` `[WARN]` を使う）。

## 品質・安定性

- 根本原因を優先し、対症療法のみで終わらせない。
- 変更後は再現可能な確認方法と未確認点を残す。
- 動作確認済みモジュールの挙動変更は、明示指示がある場合のみ行う。
- 元データは破壊せず、処理は可能な限り冪等にする。

## 報告ルール

- 重要な判断には根拠を添える。
- 思考過程の要約を必ず提示する（内部思考そのものは除く）。
- 進捗表現は次を使う: 達成(100%) / 一部達成(50-99%) / 改善中(1-49%) / 未達成(0%)。

経緯（要約）テンプレ:

```text
目的: 何を達成するか
制約: 何に従うか（ポリシー/順序/前提/ユーザー要望）
仮説: 原因候補（優先度つき）
検証: 何を見て/試して、結果がどうだったか
判断: なぜその結論・方針にしたか（根拠）
次の一手: 何をする/何をユーザーに確認するか
```

## 安全・セキュリティ

- 安全性を最優先し、疑わしい変更は実装しない。
- 機密情報（`.env`、鍵、トークン、個人情報）をコードやGit履歴に残さない。
- 既存挙動を変える変更は、明示指示と影響説明がある場合のみ行う。
- 機密値は `.env` またはシークレットマネージャーで管理する。
- 重要環境変数が未設定なら起動時に明示エラーにする。
- `.gitignore` で機密・ログ・ビルド成果物・依存ディレクトリを除外する。
- 依存関係は定期スキャンする（例: `pip-audit`, `npm audit`）。
- 外部入力は必ずバリデーションし、失敗を握りつぶさない。
- ログに機密情報を出力しない。
- インシデント関連ログは保全可能な場所へ保存する。
- 保存期間・暗号化・アクセス権はプロジェクトで定義する。

## 調査・ログ

- ログは原因再現、調査記録は再実行可能性を目的に残す。
- 機密情報を含めず、構造化・機械可読で保存する。
- ログ追加は非破壊（挙動変更なし、低遅延）で行う。
- 調査記録の保存先は `docs/調査/`、ファイル名は JST `YYYYMMDD_hhmmss_` プレフィックス必須とする。
- ログ保存先は `logs/YYYYMMDD_hhmmss_説明/` とし、`logs/` と `*.log` は Git 管理外にする。
- `audit` `state_change` `external` `business_logic` `error` `exception` `startup` `config` はサンプリング対象外。
- 同一イベントの多重記録を避ける。
- デバッグログと監査ログを混在させない。
- `password` / `token` / `api_key` / `secret` / 個人識別子 / カード番号は必ずマスクする。
- 本番は `INFO` 以上を基本とし、`DEBUG` は限定的に有効化する。

## 設計・実装・テスト

### 設計プロセス

1. 要件を明確化（5W1H）。
2. 既存実装・既存APIを確認。
3. 選択肢とトレードオフを比較。
4. 受け入れ基準を先に定義。
5. 重要判断は ADR で記録。

### 設計原則

- SOLID / DRY / KISS / YAGNI を優先する。
- 1モジュール1責務を基本とする。
- 拡張は追加で行い、既存シグネチャの破壊を避ける。
- エラーハンドリングは初期段階で設計する。

### 並行性レビュー

- TOCTOU を避ける（判定と実行を分離しない）。
- ISR は短時間・非ブロッキング、重処理はワーカー側に寄せる。
- 停止APIは冪等にする。
- 二段階停止（immediate stop / 後段 shutdown）を分離する。
- タイマー start/stop の対を明示し、停止漏れを防ぐ。

### エラーハンドリング

- 例外・エラーを握りつぶさない。
- リトライは有限回、指数バックオフ、対象限定で実装する。
- 初期化失敗は呼び出し元へ伝播し、失敗状態を明示する。

### コーディング規約

- Python: `snake_case` / 型ヒント必須 / `ruff`
- JS/TS: `camelCase` / `PascalCase` / `eslint` + `prettier` + `tsc --noEmit`
- Go: `gofmt` + `go vet`（必要なら `golangci-lint`）
- C/C++: `snake_case` + `UPPER_SNAKE_CASE` / `clang-format` / `clang-tidy`
- 公開APIはコメント・Docstringを付与する。

### Lint・テスト

- Lintエラーゼロを維持する。
- コミット前に Lint / Format / 型チェック / 関連テストを実施する。
- TDD（Red-Green-Refactor）、FIRST、AAA を推奨する。

## Chrome DevTools 安全ルール

- `take_snapshot` を使用しない（大量DOM出力による停止リスク）。
- 視覚確認は `take_screenshot`、データ取得は `evaluate_script` を使う。
- 大量データ（10,000件以上）は段階的に取得する。
- 取得対象と件数上限を先に決めてから実行する。
- 重い操作は小さな単位に分割し、結果を確認しながら進める。
